# PRD-107: Queued Delivery for Stylos Talk

- **Status:** Implemented
- **Version:** v0.66.0
- **Scope:** `themion-cli`, `themion-core`, docs
- **Author:** Tasanakorn (design intent) + Themion (PRD authoring)
- **Date:** 2026-05-08

## Summary

- Stylos `talk` currently uses target activity as a delivery guard.
- Remove that non-idle guard: a valid known-target talk must either deliver now or queue.
- Reject a valid known-target talk only when immediate delivery is unavailable and that target queue is full.
- Keep queued talk volatile and best-effort; board notes remain the durable async-work path.
- Deliver queued talk later through the same peer-message prompt path used by immediate talk.

## Goals

- Make target validity and queue capacity the main acceptance checks for `talk`.
- Stop using target activity state alone as a reason to reject a valid known-target talk.
- Keep `talk` useful for brief coordination without turning it into a durable work system.
- Deliver immediately when prompt admission is available, otherwise queue for the same target agent.
- Return a concrete `delivery_state` and `correlation_id` in `TalkReply`.
- Keep queue ownership in the CLI runtime/app-state layer, not in TUI or Stylos transport code.

## Non-goals

- Do not replace board notes or task requests for durable delegated work.
- Do not persist queued talk messages in SQLite.
- Do not guarantee delivery across process restart, crash, target-agent deletion, or queue expiry.
- Do not add threaded chat, read receipts, message acknowledgements after model processing, or exactly-once distributed delivery.
- Do not wait for the receiving agent's final natural-language answer.
- Do not add a configuration surface for queue size or TTL in the first implementation.
- Do not move Stylos coordination or queue ownership into `tui.rs`.

## Background & Motivation

### Current state

`stylos_request_talk` sends a peer message to one Themion instance and target agent. Today, the receiver validates the target and injects a peer-message prompt only when that agent is `idle` or `nap`. If the target is not idle, the request can fail with `agent_busy` unless the caller sets `wait_for_idle_timeout_ms` and the target becomes free before that wait expires.

This PRD removes target activity as a rejection guard for valid known targets. Activity state should only decide whether the message is delivered now or queued. A short peer message should not fail only because the receiver is in the middle of another turn.

### Why this is not a board-note replacement

PRD-029 and PRD-034 intentionally moved delegated asynchronous work toward durable board notes. That remains correct. Board notes are inspectable, durable, and have result state. Task requests are also still the right API when the sender needs a task id and result polling.

This PRD only adds a volatile delivery buffer for brief talk messages. A queued talk can be lost on restart or expiry. It should therefore not be used for work that must be tracked or completed later. The queue is a delivery fallback for `talk`, not a second board.

**Alternative considered:** convert busy talks into board notes automatically. Rejected: that would silently change a lightweight peer message into durable delegated work without the sender choosing that workflow.

## Design

### Acceptance rule

The receiver must accept a talk request when the target is valid and either immediate delivery or queue capacity is available.

Required behavior:

- if the target agent is not found, return `not_found`; do not create a queue for unknown targets
- if the target instance or local snapshot is unavailable, keep the existing unavailable/snapshot error behavior
- if prompt admission is available for the target, deliver immediately through the existing peer-message prompt path
- if prompt admission is unavailable, enqueue the message when the target queue has capacity
- do not reject a valid known-target talk only because the target is non-idle, running, busy, or otherwise not ready for immediate prompt injection
- after target validation, the normal rejection reason is `queue_full` when immediate delivery is unavailable and the target queue is full
- malformed payloads and invalid request shapes still reject before target acceptance

This rule is the core product change. Current activity state chooses `delivered_now` versus `queued`; it must not choose rejection by itself.

### Runtime-owned volatile inbox

The CLI runtime must own a volatile in-memory FIFO queue per target agent id in the receiving Themion process.

Queued message fields:

- `correlation_id`: generated by the receiver for every accepted talk
- `request_id`: original caller-provided request id, when present
- `from_instance`
- `from_agent_id`
- `to_instance`
- `to_agent_id`
- `message`
- `enqueued_at_ms`: Unix timestamp in milliseconds
- `expires_at_ms`: `enqueued_at_ms + 600000`

Queue constants for the first implementation:

- `MAX_QUEUED_TALK_PER_AGENT = 16`
- `QUEUED_TALK_TTL_MS = 600000` (10 minutes)

Required behavior:

- the queue is process-local and memory-only
- the queue is keyed by target `agent_id` within the receiving process
- queued messages preserve FIFO order per target agent
- queue length checks must ignore or remove expired items before deciding `queue_full`
- aggregate queue counts may be exposed through runtime-owned status snapshots
- message bodies must not be exposed through broad status snapshots
- TUI, web, headless, and Stylos paths must consume runtime-owned queue state rather than rebuilding it locally

This inbox is deliberately not the board database.

### Admission-triggered delivery

Queued talk must drain through normal runtime prompt admission, not through a direct TUI or transport shortcut.

Required behavior:

- deliver queued messages when prompt admission becomes available for the original target agent
- attempt at most one queued talk delivery per target-agent admission cycle
- inject queued talk through the same incoming-prompt path as immediate talk
- preserve the same peer-message prompt wrapper and `***QRU***` guidance used by immediate talk
- remove a queued message only after the runtime accepts it for prompt delivery
- expire stale queued talk before attempting delivery
- if prompt admission fails after dequeue selection, mark the item failed, emit an event, and do not retry that item forever
- if the target agent is deleted before delivery, fail or drop that agent's queued talk without retargeting it

Delivering one item per admission cycle keeps the first behavior predictable. Later work may add batching only if it preserves sender identity and reply clarity.

**Alternative considered:** deliver all queued messages as one combined prompt. Rejected: it blurs message boundaries and makes reply behavior harder for the model.

### Talk reply contract

`TalkReply` must report whether the talk was delivered immediately, queued, or rejected.

Required reply fields:

- `accepted: bool`
- `delivery_state: "delivered_now" | "queued" | "rejected"`
- `reason: string | null`
- `correlation_id: string | null`
- `request_id: string | null`
- `to_instance: string`
- `to_agent_id: string`
- `queue_position: integer | null`

Reply rules:

- immediate delivery returns `accepted: true`, `delivery_state: "delivered_now"`, `reason: null`, and a non-null `correlation_id`
- queued delivery returns `accepted: true`, `delivery_state: "queued"`, `reason: null`, a non-null `correlation_id`, and one-based `queue_position`
- queue-full rejection returns `accepted: false`, `delivery_state: "rejected"`, `reason: "queue_full"`, and `correlation_id: null`
- target-not-found rejection returns the existing not-found reason with `accepted: false` and `delivery_state: "rejected"`
- snapshot/unavailable and invalid-request failures keep their existing reasons with `delivery_state: "rejected"`
- do not add `queue_disabled` in the first implementation because this PRD does not add a config switch

Backward compatibility:

- existing callers that only check `accepted` still work
- new callers can inspect `delivery_state` to know whether delivery happened immediately or will happen later
- existing reason strings should remain stable where practical

### `wait_for_idle_timeout_ms` compatibility

The existing `wait_for_idle_timeout_ms` field remains accepted for compatibility, but it is no longer required to avoid activity-state rejection.

Required behavior:

- if omitted or zero, a known target that cannot accept immediate prompt injection queues immediately when capacity remains
- if positive, the receiver first waits for immediate prompt admission within the bounded wait
- if immediate admission becomes available during the wait, return `delivered_now`
- if immediate admission is still unavailable when the wait expires, queue the message when capacity remains
- if the queue is full after the wait expires, return `queue_full`
- the wait still does not wait for the receiving model's final answer

This changes talk from activity-guarded delivery to accept-or-queue delivery for valid known targets.

### Relationship to notes, tasks, and prompts

Queued delivery should strengthen brief peer coordination while preserving existing collaboration lanes.

Required behavior:

- prompt guidance must still prefer board notes for asynchronous or non-urgent delegated work
- prompt guidance must describe queued talk as volatile and best-effort
- task requests remain the API for routed work with task status/result polling
- board notes remain the API for durable work tracking and completion mentions
- queued talk uses the existing peer-message prompt wrapper and `***QRU***` guidance when injected
- the receiver must not see a different prompt shape just because the talk was queued first

The product keeps three distinct paths: talk for short coordination, tasks for task-result workflows, and board notes for durable collaboration.

### Status and observability

Queued talk should be debuggable without becoming a board UI.

Required behavior:

- emit a concise runtime event when talk is queued
- emit a concise runtime event when queued talk is delivered, expired, failed, or rejected because of limits
- include stable identifiers in events: correlation id, sender instance, sender agent id, target agent id, delivery state, and reason when present
- status snapshots may include aggregate queued-talk counts per local agent
- do not include queued talk message bodies in broad status snapshots

If the message body is shown, it should appear only through the same transcript or prompt path that already displays received talk content.

## Edge Cases

- target agent is known and immediate prompt admission is unavailable while queue has room → verify: talk returns `accepted: true`, `delivery_state: "queued"`, and a one-based `queue_position`.
- target agent is known and immediate prompt admission is unavailable while queue is full → verify: talk returns `accepted: false`, `delivery_state: "rejected"`, and `reason: "queue_full"`.
- target agent is non-idle, running, or busy for any other reason → verify: talk is not rejected solely for that activity state.
- expired queued items exist before enqueue → verify: they are removed or ignored before checking queue capacity.
- target agent becomes available after one queued talk → verify: exactly one peer-message prompt is injected and removed from the queue.
- several talks queue for one target → verify: delivery follows FIFO order across admission cycles.
- queued talk expires before prompt admission becomes available → verify: it is not injected and an expiry event is visible.
- target agent is deleted before delivery → verify: queued messages for that agent are failed or dropped without retargeting.
- process restarts with queued talk in memory → verify: queued talk is lost and no durability is claimed.
- sender uses `wait_for_idle_timeout_ms` → verify: immediate delivery still wins when available, and unavailable-after-wait behavior queues unless the queue is full.

## Implementation Notes

Implemented in v0.66.0. The landed slice removes the non-idle rejection guard, adds a process-local FIFO queue per target agent with a 16 message limit and 10 minute TTL, extends `TalkReply` with delivery state/correlation/queue position fields, and drains queued talk through the normal incoming-prompt path when admission becomes available. A follow-up in the same version also handles the admission race where a target appears idle to Stylos but becomes busy before prompt intake: peer-message intake now requeues instead of emitting `rejected: local agent busy`.

PRD-109 later renames this public surface from talk to message/inbox terminology and removes the wait-for-idle option. Treat this PRD as the historical contract for the queued-delivery behavior that first shipped under the old `talk` names.

## Migration

This is a backward-compatible behavior change for valid `stylos_request_talk` calls.

Existing callers that only check `accepted` continue to work. They should see more accepted responses because known targets now queue talk when immediate delivery is unavailable instead of rejecting with `agent_busy`. Callers that need to know whether delivery was immediate should read `delivery_state`.

No database migration is required because queued talk is memory-only.

The repository version was bumped to `v0.66.0` for this implementation.

## Testing

- send talk when immediate prompt admission is available → verify: current immediate peer-message injection still works and returns `delivery_state: "delivered_now"`.
- send talk to a known target when immediate prompt admission is unavailable and its queue has room → verify: response is accepted, marked `queued`, and includes `queue_position`.
- fill the per-agent queue with 16 unexpired messages → verify: the next non-immediate talk is rejected with `queue_full`.
- enqueue 16 messages, expire them, then enqueue another message → verify: expired messages do not cause `queue_full`.
- let prompt admission become available for the target → verify: one queued talk is injected through the normal incoming-prompt path.
- enqueue multiple talks for one target → verify: delivery preserves FIFO order across admission cycles.
- advance past `QUEUED_TALK_TTL_MS` or simulate stale queued talk → verify: stale messages expire and are not injected.
- restart the process during queued talk → verify: queued talk does not survive and docs do not claim durability.
- send talk with positive `wait_for_idle_timeout_ms` while target becomes available during the wait → verify: response is `delivered_now`.
- send talk with positive `wait_for_idle_timeout_ms` while target remains unavailable but queue has room → verify: response is `queued`.
- run `cargo check -p themion-cli` → verify: default build compiles.
- run `cargo check -p themion-cli --features stylos` → verify: Stylos-enabled build compiles.
