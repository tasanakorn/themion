# PRD-001: Config File + REPL Verbose Feedback

- **Status:** Proposed
- **Version:** v0.1.0
- **Scope:** `themion-cli` (config loading, REPL output); `themion-core` (agent event hooks)
- **Author:** Tasanakorn (design) + Claude Code (PRD authoring)
- **Date:** 2026-04-18

## 1. Goals

- Load configuration from an XDG-standard path on startup, and generate a self-documenting default file on first run.
- Surface progress in REPL mode so users see when the agent is thinking, calling a tool, or receiving a tool result.
- Keep print mode (single-turn invocation with a CLI arg) silent apart from the final answer, so it remains scriptable.

## 2. Non-goals

- No GUI config editor, TUI picker, or interactive setup wizard.
- No persistent REPL history across sessions (readline history, up-arrow recall).
- No structured logging framework or log levels; verbose output is plain `println!` to stdout.
- No reload-on-change or hot-reload of the config file.

## 3. Background & Motivation

Today, every user must export three environment variables before launching `themion`, and a missing `OPENROUTER_API_KEY` triggers a raw `.expect()` panic. During a multi-step agent run, the REPL is silent from the moment the user presses Enter until the final text is produced â€” there is no indication that a tool was called, nor how many round-trips the LLM took. Users report the session feeling "frozen" and sometimes Ctrl-C out of healthy runs.

Introducing a config file (XDG path, auto-created with comments) removes the env-var friction for first-time setup, and adding lightweight status lines in REPL mode makes multi-step runs legible without adding framework complexity.

### 3.1 Current state

- `crates/themion-cli/src/main.rs` lines 7â€“12 read all three config values directly from environment variables. `OPENROUTER_API_KEY` missing triggers `.expect(...)` which panics with a stack trace rather than a friendly exit.
- `OPENROUTER_MODEL` defaults to `minimax/minimax-m2.7`; `SYSTEM_PROMPT` defaults to a generic assistant prompt.
- No TOML parser, no `dirs` crate. `serde` with the `derive` feature is already a workspace dependency.
- `Agent::run_loop` in `crates/themion-core/src/agent.rs` has no hook points: it pushes the user message, calls the client, executes tool calls in a `for` loop, and returns the final content string with no intermediate output.
- Four tools are dispatched by name: `read_file`, `write_file`, `list_directory`, `bash` (see `crates/themion-core/src/tools.rs`).

## 4. Design

### 4.1 Config file location and format

The config file lives at `$XDG_CONFIG_HOME/themion/config.toml`, falling back to `~/.config/themion/config.toml` via `dirs::config_dir()`. On Linux both paths resolve identically for typical users; on macOS `dirs::config_dir()` returns `~/Library/Application Support`, which is acceptable for this release.

The file format is TOML with three optional top-level keys matching today's env vars:

```toml
# themion config file
# Uncomment and edit to set defaults; env vars override these values.

# api_key = "sk-or-v1-..."
# model = "minimax/minimax-m2.7"
# system_prompt = "You are a helpful AI assistant with access to tools."
```

On first run, if the file does not exist, `themion-cli` creates the parent directory (`mkdir -p` equivalent) and writes the template above with all three fields commented out. The file is never overwritten once it exists.

**Alternative considered:** JSON or YAML for the config file. Rejected: TOML is idiomatic for Rust CLIs (Cargo, rustup) and supports inline comments without extra parser features.

**Alternative considered:** a `themion init` subcommand to create the config explicitly. Rejected: the first-run auto-create has zero ceremony and the file is trivially regenerated by deleting it.

### 4.2 Config resolution order

Precedence, highest wins:

1. Environment variable (`OPENROUTER_API_KEY`, `OPENROUTER_MODEL`, `SYSTEM_PROMPT`).
2. Value present in `config.toml`.
3. Built-in default (only for `model` and `system_prompt`; `api_key` has no default).

If after resolution `api_key` is still empty, `themion-cli` prints a friendly one-liner to stderr pointing at the config file path and the env var name, then exits with code 1. No panic.

**Alternative considered:** config file wins over env vars. Rejected: env vars are the conventional override channel for short-lived experiments (e.g. `OPENROUTER_MODEL=... themion ...`); users expect them to take precedence.

### 4.3 Config loader module

A new module `crates/themion-cli/src/config.rs` exposes:

```rust
pub struct Config {
    pub api_key: String,
    pub model: String,
    pub system_prompt: String,
}

impl Config {
    pub fn load() -> anyhow::Result<Self> { ... }
}
```

`Config::load` performs: resolve config dir â†’ ensure file exists (create with template if absent) â†’ parse TOML into an `Option<...>`-typed intermediate â†’ layer env vars on top â†’ fill defaults â†’ validate `api_key` non-empty.

Dependencies added:

- `toml = "0.8"` â€” workspace dep (may later be reused by other crates).
- `dirs = "5"` â€” cli-only dep (path resolution is a CLI concern, not a library concern).

### 4.4 Verbose feedback in REPL

A new `verbose: bool` field is added to `Agent`. A constructor variant `Agent::new_verbose(client, model, system_prompt)` sets it to `true`; the existing `Agent::new` keeps `verbose: false` for backwards compatibility.

Inside `run_loop`, three status lines are emitted to stdout via `println!` when `self.verbose == true`:

| Hook point                         | Output                                 |
| ---------------------------------- | -------------------------------------- |
| Before each `chat_completion` call | `[thinking...]`                        |
| Before executing each tool call    | `[calling tool: {name}]`               |
| After each tool call returns       | `[tool result received]`               |

Print mode (`main.rs` when CLI args are present) continues to use `Agent::new`, so no status lines appear. REPL mode uses `Agent::new_verbose`.

**Alternative considered:** an event channel (`tokio::sync::mpsc`) where `run_loop` sends `AgentEvent` enum variants and the CLI consumes them. Rejected: adds a concurrent task and a public event type for a feature that only needs three stdout lines; the `bool` flag is smaller and symmetric to how `OpenRouterClient` already treats the model as plain config.

**Alternative considered:** a callback closure passed to `run_loop`. Rejected: adds a lifetime parameter or a `Box<dyn Fn>` to the hot path; ergonomic cost outweighs the flexibility for a single consumer.

### 4.5 Output styling

Status lines go to **stdout**, not stderr, so they interleave cleanly with the assistant response in a user's terminal and with any pager. No ANSI color codes in this release â€” the lines are plain ASCII so they pipe through `tee`, `less`, and log capture unchanged.

## 6. Edge Cases

- **Config dir unavailable** (e.g. `$HOME` unset on a locked-down CI runner): `dirs::config_dir()` returns `None` â†’ `Config::load` skips the file layer entirely and relies on env vars only. No error.
- **Config file exists but is malformed TOML**: return an `anyhow::Error` pointing at the file path and the parser message. Do not overwrite it â€” the user probably wants to fix it by hand.
- **Config file has unknown top-level keys**: silently ignore (serde `#[serde(default)]` on each field, no `deny_unknown_fields`). This keeps old binaries forward-compatible with newer config files.
- **Empty string `api_key` in config file**: treated the same as missing; env var or friendly error.
- **Very long tool output**: the `[tool result received]` line is a fixed-size ack; the full result still flows to the LLM, not to the terminal. No risk of scroll-flooding from verbose mode itself.
- **Interleaving with stdin prompt**: status lines print before the agent returns, so the next `> ` prompt only renders after `run_loop` completes. No race.

## 7. Migration

Existing users who rely on env vars see zero behavior change: env vars still take precedence, and an empty config file (the auto-generated one has every field commented out) is equivalent to no file at all. On first run, users notice a new file at `~/.config/themion/config.toml`; they may either ignore it or uncomment `api_key` to drop the export from their shell profile.

No code outside `themion-cli` and `themion-core` is affected. `themion-core`'s public API grows by one constructor (`Agent::new_verbose`); existing `Agent::new` callers keep working unchanged.
