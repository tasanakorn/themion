# PRD-047: Prefer `note_slug` in User-Facing Board Note Chat Events

- **Status:** Implemented
- **Version:** v0.29.2
- **Scope:** `themion-cli`, docs
- **Author:** Tasanakorn (design) + Themion (PRD authoring)
- **Date:** 2026-04-24

## Summary

- Board notes already have two identifiers: canonical UUID `note_id` and human-friendly `note_slug`.
- User-facing chat-panel events still expose long UUIDs in places such as `created board note in db note_id=...`.
- Change user-facing note event text to prefer `note_slug` where a note identifier is displayed to the operator.
- Keep UUID `note_id` as the canonical machine identifier for tools, persistence, and internal lookups.
- Keep prompt/tool payloads explicit enough for machines; this PRD is about human-facing chat-panel display and nearby event wording.
- Preserve the ability to include UUIDs in debug or structured contexts when they are genuinely needed for diagnosis.

## Goals

- Make chat-panel board-note events easier for humans to scan by displaying `note_slug` instead of raw UUID `note_id` as the primary visible note identifier.
- Apply the display preference consistently to user-facing note lifecycle events in the TUI chat panel, including note creation, note intake, note injection, and done-mention events where note identity is shown.
- Preserve UUID `note_id` as the canonical durable identifier used by `board_read_note`, `board_move_note`, `board_update_note_result`, persistence helpers, and machine-facing payloads.
- Avoid changing the SQLite schema or the two-identifier note model introduced by earlier PRDs.
- Keep logs and protocol-ish machine-facing payloads unambiguous where UUID identity is required.
- Update docs so the distinction is clear: `note_slug` is preferred for user display, while `note_id` remains canonical for tool calls and storage.

## Non-goals

- No replacement of UUID `note_id` as the canonical note identifier.
- No schema migration or identifier regeneration.
- No change to `board_*` tool input contracts that currently require canonical UUID `note_id`.
- No removal of `note_id` from structured tool results, note injection metadata, or other machine-oriented contexts.
- No redesign of board columns, note delivery, idle-time injection, done mentions, or Stylos note transport.
- No broad TUI redesign for a full board UI.
- No requirement to make slugs globally routeable protocol identifiers beyond their current human-friendly companion role.

## Background & Motivation

### Current state

PRD-030 introduced the current two-identifier note model:

- `note_id` is a UUID and remains the canonical machine-facing durable identifier
- `note_slug` is a globally unique human-friendly companion identifier for logs, debugging, and future UI display

The architecture docs also describe durable notes as storing both canonical UUID `note_id` and globally unique human-friendly `note_slug`.

However, some user-facing TUI chat-panel events still show the UUID as the primary identifier. A current example is:

```text
󰀂 created board note in db note_id=7986c159-39af-44f4-af3f-d35f07dfcaf4 from=vm-02:130390
```

That is technically correct, but it is not ideal for operator-facing display. UUIDs are long, visually noisy, and harder to relate to the human-readable note names already generated by the system.

### Why `note_slug` should be primary in chat-panel display

The chat panel is primarily an operator-facing transcript. When it reports that a note was created, received, injected, or completed, the most useful visible identifier is the human-friendly slug.

Displaying the slug first improves usability:

- shorter and easier to scan in the conversation pane
- aligns with the reason `note_slug` exists
- reduces UUID noise in status-like event lines
- gives humans a stable readable handle for later discussion or inspection

**Alternative considered:** keep UUIDs everywhere because they are canonical. Rejected: canonical machine identity does not need to be the primary human display identity in the chat panel, especially when a unique human-friendly companion already exists.

### Why UUID `note_id` must remain canonical internally

The slug display preference should not weaken the durable note contract. Tools and persistence still need canonical UUID identity because it is stable, unambiguous, and already used by note operations.

The right split is therefore:

- user-facing chat-panel note identity: prefer `note_slug`
- machine-facing note identity and tool inputs: keep `note_id`
- detailed/debug contexts: may include both when useful

**Alternative considered:** switch tool inputs to accept only `note_slug`. Rejected: that would be a broader API change, would weaken the canonical role of UUID identity, and is unnecessary for the display issue.

### Why this is a patch-level UX correction

The note system already has the required data. The requested change does not add a new board capability or alter note lifecycle behavior. It only changes which existing identifier is shown first in user-facing chat-panel event text.

That makes the change a patch-level polish/correctness adjustment for the existing notes UX.

**Alternative considered:** defer this until a future full board UI. Rejected: the current chat-panel event text is visible today and can be improved independently with a narrow change.

## Design

### Prefer `note_slug` in user-facing chat-panel note event lines

When the TUI emits a chat-panel event line that identifies a board note for the user, it should display `note_slug` as the primary note identifier.

Normative direction:

- replace primary `note_id=<uuid>` display in user-facing note events with `note_slug=<slug>` when the slug is available
- apply this to note creation, note intake, note injection, and done-mention chat-panel events where the event is intended for human/operator consumption
- keep event wording concise and status-like
- avoid showing both identifiers by default in ordinary chat-panel events unless the event would otherwise become ambiguous

For the current reported example, the desired direction is a line shaped like:

```text
󰀂 created board note in db note_slug=create-prd-for-displaying-note-s-45a02b99 from=vm-02:130390
```

The exact slug value depends on the created note, but the visible key should prefer `note_slug`.

**Alternative considered:** show `note_slug=<slug> note_id=<uuid>` in every event. Rejected: including both everywhere preserves most of the visual noise this PRD is trying to remove. Both may still be appropriate in debug-specific output.

### Keep `note_id` in machine-facing and tool-facing contexts

The display change should not alter structured note contracts.

Normative direction:

- keep `note_id` in `board_*` tool inputs and outputs where it is part of the canonical note contract
- keep `note_id` in note injection metadata that the model may need for direct `board_*` mutations, unless a later PRD changes tool input support
- keep persistence and internal event routing keyed by UUID `note_id`
- do not infer note identity from `note_slug` in internal mutation paths unless already supported elsewhere

This preserves the existing storage and tool API semantics.

**Alternative considered:** remove `note_id` from all model-visible contexts and rely on slug. Rejected: current board mutation tools are UUID-oriented, and hiding UUIDs from machine contexts would make tool use harder unless the tool contracts change too.

### Define a user-facing versus machine-facing boundary

The implementation should distinguish display strings from structured contracts.

User-facing chat-panel examples include:

- receiver-side `created board note in db ...`
- inbound `Board note intake ...` notices shown in the conversation pane
- `Board note injection ...` notices shown to the local operator
- `Board done mention ...` notices shown in the conversation pane

Machine-facing or structured contexts include:

- SQLite rows and persistence helpers
- `board_*` tool JSON results
- internal note selection and mutation calls
- prompt wrappers that must provide enough metadata for the model to operate tools correctly
- protocol/queryable payloads that use canonical note identity

Normative direction:

- prefer `note_slug` in the first group
- preserve `note_id` in the second group
- when a line serves both purposes, prefer a structured payload or include both only if needed for correctness

**Alternative considered:** classify every chat-panel event as machine-facing because the model can read transcript text. Rejected: the chat panel is an operator-facing UI, and the model has proper tool/prompt metadata paths for machine actions.

### Fall back clearly if a slug is unavailable

`note_slug` should normally be available for durable notes, but fallback behavior should be explicit for defensive robustness.

Normative direction:

- if a note event genuinely lacks a slug, fall back to `note_id=<uuid>` rather than hiding note identity
- do not synthesize a misleading slug in display code
- prefer fixing missing slug propagation at the source when the stored note has a slug

This keeps display robust without weakening the storage contract.

**Alternative considered:** suppress note identity entirely when slug is missing. Rejected: a note event without any identifier is less useful for debugging and operator follow-up.

### Update docs to state the display convention

The existing docs already describe the two-identifier model. They should also describe the display convention introduced here.

Normative direction:

- update architecture/runtime docs where they show or describe chat-panel note events
- state that chat-panel/operator-facing events prefer `note_slug`
- keep docs clear that `note_id` remains canonical for tools and storage
- update examples that currently show `note_id=<uuid>` in user-facing note event lines

**Alternative considered:** change code only and leave docs with UUID examples. Rejected: docs are the contract for these observable event strings and should match the intended UI behavior.

## Edge Cases

- a newly created note has both UUID `note_id` and `note_slug` → verify: the chat-panel creation event displays `note_slug=<slug>` rather than `note_id=<uuid>`.
- a note-created event somehow lacks a slug → verify: the chat-panel event falls back to a clear `note_id=<uuid>` display instead of omitting identity.
- a user-facing intake event is emitted for an inbound note → verify: the visible note identifier is the slug when available.
- an idle-time note injection event is displayed → verify: the visible chat-panel event uses the slug while the injected prompt metadata still includes enough canonical note information for tool calls.
- a done-mention note is created → verify: the user-facing completion event displays the done mention's slug and still preserves original-note reference metadata where machine logic needs it.
- code currently parses `note_id=` out of a chat-panel event line → verify: the implementation does not silently break that logic; either keep parsing on an internal machine string or replace it with structured data.
- a `board_read_note` or `board_move_note` call is needed after seeing a slug-only chat event → verify: the model still receives canonical `note_id` through tool results, injection metadata, or an explicit board lookup path rather than relying only on chat-panel prose.
- debug output intentionally needs UUID identity → verify: debug-specific or structured output may include `note_id` without violating the ordinary chat-panel display convention.

## Migration

This is a display-only patch over the existing note identity model.

Expected rollout shape:

- no database migration
- no change to existing note rows
- no change to canonical `note_id` semantics
- update only user-facing chat-panel event strings and related docs
- keep fallback behavior for missing slugs
- preserve or refactor any internal logic that currently depends on parsing UUIDs from display strings before changing the visible text
